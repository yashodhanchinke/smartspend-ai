================================================================================
2) BACKEND APIs — USER-CONTROLLED INPUTS (REQUEST PAYLOAD FIELDS)
================================================================================

Important note
- The backend endpoints accept JSON payloads. In this project, the mobile app supplies these payloads.
- The “manual user input” behind these payloads comes from the fields listed in section 1 (Reports filters, Profile email, etc.).

Backend file: backend/server.js

Endpoint: POST /api/monthly-report
User-controlled payload keys (from backend/server.js destructuring):
- month (string "YYYY-MM") — derived from ReportsScreen period selection
- range_days (number) — fixed 30 in ReportsScreen for last-30-days report
- custom_start_date (string "YYYY-MM-DD") — from ReportsScreen filter start date
- custom_end_date (string "YYYY-MM-DD") — from ReportsScreen filter end date
- account_ids (string[]) — from ReportsScreen selected accounts
- category_ids (string[]) — from ReportsScreen selected categories
- send_email (boolean) — controlled by “Email” action in ReportsScreen
- email_to (string) — taken from Profile email (ProfileScreen input)
- report_label (string) — generated by client code (not typed)
- filters (object) — generated by client code (snapshot of selected filter settings)

Endpoint: POST /api/email-report
User-controlled payload keys:
- email_to (string) — comes from Profile email (ProfileScreen input)
- report_id / filename — selected report item (user tap selection, not typed)

Endpoint: POST /api/notifications/register-device
Payload keys:
- expo_push_token, push_permission_status, language_mode
These are system-generated (permissions + device) in NotificationContext; not manually typed.

Endpoint: POST /api/notifications/generate
Payload keys:
- force (boolean) — driven by UI refresh actions; not typed.

Endpoint: POST /api/sms/analyze
Payload keys:
- sender, message
These come from SMS listener; not typed by the user.

3) DATABASE SCHEMA — USER-MODIFIABLE FIELDS (WHAT USERS END UP POPULATING)
================================================================================

These are the key DB columns that are populated by user input through the screens above.
Schema reference: supabase/schema_reference.sql

Table: public.profiles
- user input: name, email (ProfileScreen, RegisterScreen)

Table: public.accounts
- user input: name, type, balance (initial), is_default, color
- system/auto: icon

Table: public.categories
- user input: name, type, color
- system/default: icon is "tag" unless set elsewhere
- system: many default categories are auto-inserted by util/seedCategories.js

Table: public.transactions
- user input: type, title (optional), amount, date, time, account_id, category_id OR to_account_id (for transfer)

Table: public.budgets + public.budget_categories
- user input: name, amount, period, mode, budget_type, notes, color, category selections

Table: public.goals
- user input: title, target_amount, start_date, end_date, color
- system: current_amount starts at 0 unless updated elsewhere

Table: public.loans
- user input: type, name, amount, description, start_date, end_date
- system: status defaults to pending; settlement fields are filled when user settles

Table: public.recurring_transactions
- user input: title, amount, type, period, next_run, account_id, category_id

Table: public.labels
- user input: name, color

Table: public.user_reports
- user input: indirect (filters + profile email), through ReportsScreen actions
- system: storage_path, filename, AI texts, timestamps
